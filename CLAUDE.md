# INDICATE Data Dictionary

## Project Purpose

This repository hosts the **INDICATE Data Dictionary** content as JSON files, and a static HTML catalog (GitHub Pages) to browse concept sets, concepts, and projects without needing the full Shiny application.

## Build Pipeline

### Step 1: Resolve concept sets (`resolve.py`)

Resolves concept set expressions by expanding descendants and mapped concepts using OMOP vocabulary tables stored in a DuckDB database.

**Prerequisites**: `pip install duckdb` and a DuckDB database containing OMOP vocabulary tables (`concept`, `concept_ancestor`, `concept_relationship`).

**Usage**:
```bash
python3 resolve.py --db /path/to/ohdsi_vocabularies.duckdb
```

**Algorithm** (matches R function in `data-dictionary/R/fct_duckdb.R`):
1. Partition expression items into included and excluded
2. For each partition, expand with descendants (`includeDescendants`) and mapped concepts (`includeMapped`)
3. Final resolved set = included ‚àí excluded
4. Fetch concept details from the `concept` table

**Output**: `concept_sets_resolved/{id}.json` for each concept set.

### Step 2: Build data files (`build.py`)

Aggregates all JSON files into data files used by the static site.

**Usage**:
```bash
python3 build.py
```

**What it does**:
1. Loads all `concept_sets/*.json` files
2. Loads all `projects/*.json` files
3. Loads all `concept_sets_resolved/*.json` files (if the directory exists)
4. Loads `units/unit_conversions.json` and `units/recommended_units.json`
5. Loads `mapping_recommendations/mapping_recommendations.json`
6. Computes a content hash for cache-busting
7. Produces three output files:
   - `docs/data.json` ‚Äî compact JSON with all data (`conceptSets`, `projects`, `resolvedConceptSets`, `unitConversions`, `recommendedUnits`, `mappingRecommendations`, `dataVersion`, `dataHash`)
   - `docs/data_inline.js` ‚Äî same data wrapped as `const DATA={...};` for direct `<script>` inclusion
   - `docs/resolved_concept_ids.json` ‚Äî unique concept IDs from resolved sets (used by DuckDB loader for filtering)

### Full rebuild

```bash
python3 resolve.py --db /path/to/ohdsi_vocabularies.duckdb
python3 build.py
```

If you don't have a DuckDB vocabulary database, you can skip `resolve.py` and just run `build.py` ‚Äî resolved concept sets will be empty but the site will still work (only the "Expression" tab will show concepts, not the "Resolved" tab).

## GitHub Pages Catalog

The static site is a **single-page application (SPA)** with hash-based routing. All pages are rendered within `index.html`; secondary HTML files (`projects.html`, `settings.html`, etc.) are redirects for backwards compatibility.

### Pages

| Route | Module | Description |
|---|---|---|
| `#/concept-sets` | `ConceptSetsPage` | Browse, search, filter concept sets; detail view with expression, resolved concepts, comments, statistics, review |
| `#/mapping-recommendations` | `MappingRecommendationsPage` | View and edit mapping recommendations |
| `#/projects` | `ProjectsPage` | Browse projects, view linked concept sets |
| `#/settings` | `SettingsPage` | Dictionary settings: unit conversions, recommended units, vocabulary tables |
| `#/dev-tools` | `DevToolsPage` | SQL editor (Ace + DuckDB-WASM) and ERD visualization (vis-network) |
| `#/documentation` | `DocumentationPage` | Built-in bilingual documentation |

### Architecture

```
index.html (SPA shell)
  ‚îú‚îÄ‚îÄ app.css               (all styles)
  ‚îú‚îÄ‚îÄ data_inline.js        (generated data)
  ‚îú‚îÄ‚îÄ app.js                (window.App ‚Äî shared state & utilities)
  ‚îú‚îÄ‚îÄ duckdb-loader.js      (window.VocabDB ‚Äî DuckDB-WASM vocabulary DB)
  ‚îú‚îÄ‚îÄ router.js             (hash-based SPA router)
  ‚îú‚îÄ‚îÄ concept-sets.js       (ConceptSetsPage)
  ‚îú‚îÄ‚îÄ projects.js           (ProjectsPage)
  ‚îú‚îÄ‚îÄ mapping-recommendations.js (MappingRecommendationsPage)
  ‚îú‚îÄ‚îÄ settings.js           (SettingsPage)
  ‚îú‚îÄ‚îÄ general-settings.js   (GeneralSettingsPage ‚Äî vocabulary import)
  ‚îú‚îÄ‚îÄ dev-tools.js          (DevToolsPage)
  ‚îú‚îÄ‚îÄ documentation.js      (DocumentationPage)
  ‚îî‚îÄ‚îÄ spa-init.js           (boots the app: registers routes, starts router)
```

- **`app.js`** exposes a global `window.App` object (revealing module pattern) with shared state (`conceptSets`, `projects`, `lang`, `userConceptSets`, `userProjects`), utilities (`escapeHtml`, `showToast`, `renderMarkdown`, `statusBadge`, `translateDOM`, etc.), profile management, data loading/merging, and shared event handlers.
- **`router.js`** provides hash-based routing (`#/concept-sets`, `#/projects?id=1`, etc.) with query parameter support.
- **`spa-init.js`** registers all page routes, wires up language change callbacks, and boots the app.
- **`duckdb-loader.js`** loads OHDSI Athena vocabulary files (CSV/Parquet) into an in-memory DuckDB-WASM database, persisted to IndexedDB. Exposes `window.VocabDB`.
- Each page module is an IIFE exposing `show(query)`, `hide()`, and optionally `onLanguageChange()`.
- Navigation uses hash links (`#/concept-sets`). Language preference and user data persist via `localStorage`.
- Deep linking: `#/concept-sets?id=123` auto-opens the concept set detail view.
- **Local data editing**: Users can create/edit concept sets and projects locally (stored in `localStorage`), with modified items overriding repo data.

### Review & "Propose on GitHub" workflow

The Concept Sets page includes a review feature:
1. Users can submit reviews (status + comments) via the Review tab
2. Reviews are stored in `App.sessionReviews` (persisted to `localStorage`)
3. After submitting a review, a "Propose on GitHub" button appears, which:
   - Copies the full concept set JSON (with the review included) to the clipboard
   - Opens `https://github.com/indicate-eu/data-dictionary/edit/main/concept_sets/{id}.json` in a new tab
   - The user can then paste the JSON, commit to a new branch, and open a PR

### DuckDB-WASM Vocabulary Database

Users can load OHDSI Athena vocabulary files directly in the browser:
- **General Settings page** (`#/settings` ‚Üí General Settings): import CSV or Parquet vocabulary files
- Files are loaded into DuckDB-WASM and persisted to IndexedDB for subsequent visits
- **Dev Tools page** (`#/dev-tools`): SQL editor (powered by Ace) to query the loaded vocabulary database, plus an ERD visualization (powered by vis-network)
- Tables loaded: `CONCEPT`, `CONCEPT_ANCESTOR`, `CONCEPT_RELATIONSHIP`, `CONCEPT_SYNONYM`, `RELATIONSHIP`, `VOCABULARY`, `DOMAIN`, `CONCEPT_CLASS`, optionally `DRUG_STRENGTH`

## Important Workflow Rules

- **After any change to source data files** (`concept_sets/`, `projects/`, `concept_sets_resolved/`, `units/`, `mapping_recommendations/`), you MUST run `/build-catalog` to regenerate `docs/data.json` and `docs/data_inline.js`. Without this step, the GitHub Pages site will not reflect the changes.
- **Never edit `docs/data.json`, `docs/data_inline.js`, or `docs/resolved_concept_ids.json` by hand** ‚Äî they are generated files. Always edit the source JSON and rebuild.
- **ID counters** (`id_counters.json`): contains `nextConceptSetId` and `nextProjectId` ‚Äî monotonic counters that never decrease. `build.py` validates and auto-corrects them at build time. When creating a new concept set or project JSON file manually, increment the relevant counter. Never reuse a deleted ID.

## Key Conventions

- **Bilingual**: All concept sets have English and French translations in `metadata.translations`
- **OMOP CDM**: Concepts follow the OMOP Common Data Model vocabulary (SNOMED, LOINC, RxNorm, etc.)
- **Athena links**: Concept IDs link to `https://athena.ohdsi.org/search-terms/terms/{conceptId}`
- **Categories**: Concept sets are grouped by category/subcategory from metadata translations
- **Versioning**: Concept sets use semantic versioning (`version` field)
